SCOPE/Hustler Demo

This document describes how to run SCOPE/Hustler using the PC-based mainframe emulator DtCyber.  It specifically focuses on running a kiosk-style demo.  The demo is really a running copy of SCOPE/Hustler--not a mockup.

Installation

You should have three zip archives of files from MRR containing:

  • servers:  The DtCyber CDC 6000 hardware emulator and the frend2 front-end (FREND) emulator.  This acts as a telnet server running on port 6500.
  • media:  Disk and tape image files that are used to emulate CDC disk drives and tape drives.
  • client:  The telexpt client program, which acts as a terminal emulator.  This is an optional component.  Also included is a copy of the Win32 program scredit2, a partial SCREDIT emulator.

You will also need a decent PC running a 32-bit, or better yet, 64-bit version of Windows.  (On the same hardware, the 64-bit version of DtCyber is about twice as fast as the 32-bit version.)  Note: DtCyber will consume 100% of your CPU.  Don't run it on a computer that is being used for other purposes unless you have multiple CPUs.

Unzip all files into a directory.  I suggest C:\DtCyber.  Some of the zip files contain directory names, so use an unzip program that respects directory names (most do).

Running the clients

For a SCOPE/Hustler demo to be interesting, the machine should be running some jobs.  I have written a simple Java-based terminal emulator, telexpt, which runs an interactive session via a script.  (I tried to use the well-known program Expect, but it simply does not work under Windows with Telnet.)

If your computer has two monitors, you can run the client on the same computer that is running DtCyber.  telexpt does not require much CPU, so it won't significantly harm DtCyber's performance.

If your computer does not have two monitors, you will want to run the clients on a separate computer, so the session windows do not overlap the Cyber console window.  If they do overlap, no harm is done--but the demo will be less interesting.

telexpt requires the Java Runtime.  Install it.  Make sure that the Java runtime's bin directory is in your path.

Open a command prompt and change directory to DtCyber\client.  Run startClients.bat.  This will run two clients.

Problems

Recent stress-testing has revealed some problems with interactive usage.  I probably won't have time to fix them before the demo.

It appears that some software component--probably frend2--cannot cope with many (or even more than two?) simultaneous interactive sessions.  So, don't try more than two sessions.  frend2's console allows one special interactive session that does not involve telnet:  press F2 in the frend2 window.  This third session does not seem to interfere with the telnet sessions, so it is a way of having a third session.

MANAGER sometimes aborts.  To fix this without redeadstarting:

  • Exit frend2 (hit F10 in the frend2 window).
  • Run frend2.exe from the DtCyber directory.
  • Run MANAGER again (1.X MANAGER,NOLOAD.)

Sometimes you can simply run MANAGER again without bouncing frend2.
